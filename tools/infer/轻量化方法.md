### 整体推理流程

整幅 LAS 点云
  -> 切成多个 tile 大块
  -> 每个 tile 内做 voxel 降采样/网格化
  -> 再切成多个 fragment 小片送进模型
  -> 模型输出每个点的类别
  -> 把所有 tile/fragment 的结果合回原 LAS

### tile

    tile:就是把整幅LAS按XY平面切成一块一块的大区域；``   --tile-size 40     --tile-stride 40   ``
**tile-size:**
        意思是每个 tile 是 40m x 40m，相邻 tile 不重叠。
        如果一整幅点云很大，不能一次送进模型，就必须分 tile 推理。

**影响：**
        ``       tile-size 大：块少，上下文更完整，但单块点数多，可能很慢         tile-size 小：块多，单块更快，但边界更多，拼接误差更明显       ``

**tile-stride:**
        tile-stride 是 tile 移动步长。
        ``       --tile-size 40         --tile-stride 40       ``
        表示不重叠切块。
        ``       --tile-size 40         --tile-stride 20       ``
        表示每个 tile 重叠一半。
        重叠越多，边界效果可能更稳，但推理次数变多，速度会明显变慢。

### merge-mode

    他决定多个tile结果怎么合并。``   --merge-mode plain：普通不重叠推理，最快，但是边界容易有误差。     --merge-mode halo：带上下文推理，比如核心 tile 是 40m x 40m，但推理时额外看周围 context-margin 米，最后只保存核心区域结果。这个很适合输电线路。     --merge-mode overlap：重叠推理，同一点可能被多个tile预测，然后做概率平均，最稳，但最慢。   ``

**context-margin**
        context-margin 就是 halo 模式里的上下文范围。
        ``       --tile-size 40         --context-margin 10       ``
        核心区域是 40m x 40m，但实际推理时会看周围额外 10 米：
        context-margin 大：边界更稳，但每块点更多，速度更慢
        context-margin 小：速度快，但边界改善有限

### voxel

    voxel:三维体素；3D网络。
    点云太密时，很多点非常接近。voxel 会把空间划成小格子，每个格子选一个代表点，从而减少点数。``   --pre-voxel-size     --grid-size   ``
**pre-voxel-size**
    推理前对每个 tile 先做一次降采样。
    例如：
    ``   --pre-voxel-size 0.05：表示用5cm的体素网格先压缩点云。   ``
    怎么做：
    比如原来的一个tile：1000000points
    经过降采样后变为300000points
    模型只推理这30万点，然后结果映射到原始100万点。
    ``   0      不做预降采样，最慢，理论保留最多细节     0.05   推荐，速度和精度平衡     0.06   更快一点，通常还能接受     0.08   更快，但电线/绝缘子可能受影响     0.10   风险较大，细线结构可能变差   ``
**grid-size**
    grid-size 是 Pointcept 测试 pipeline 里的体素化参数，来自配置文件。
    它影响模型内部测试时的 GridSample。一般不要随便改，因为它跟训练时的数据处理方式有关。
    如果训练配置里用的是某个固定 grid_size，推理最好保持一致。

    ``   优先调 pre-voxel-size     谨慎调 grid-size   ``

### fragment

    fragment 是 tile 里面进一步切出来、真正送进模型的小片。
    即使一个 tile 是 40m x 40m，里面可能有几十万甚至上百万点，模型也不能一次吃完，所以脚本会根据 point_max 把 tile 切成多个 fragment。``fragments: 18``
    表示这个tile被分成了18个模型的输入片段
    影响：
    fragment 越多：推理越慢
    fragment 越少：推理更快，但单次显存占用更高

### point-max

    point-max 控制每个 fragment 最多包含多少点`--point-max 120000`
    表示每个 fragment 最多约 12 万点。

    point-max 小：fragment 多，速度慢，但显存安全
    point-max 大：fragment 少，速度快，但显存占用高

    如果显存没满，可以适当增大：

### fragment-batch-size

    这个控制一次送几个 fragment 到 GPU。``   batch-size 小：显存低，但 GPU 利用率可能不高     batch-size 大：速度可能更快，但显存更高   ``
    这个通常不影响精度，只影响速度和显存。

### min-tile-points

    这个表示点数太少的 tile 直接跳过。
    比如：`--min-tile-points 1024`如果某个 tile 只有几百个点，就不推理了，默认给背景或保持默认结果。
    影响：

    值大：跳过更多稀疏区域，速度快，但可能漏掉少量目标
    值小：更完整，但慢一点

### disable-flash

    `--disable-flash`
    表示关闭 flash attention。
    如果 Linux 上 flash attention 可用，一般不要加这个参数。开启 flash 通常更快。

## 更深层的轻量化方法

上面的 tile、voxel、fragment、batch size 主要属于推理参数优化，不需要重新训练模型。
如果继续往深一点做轻量化，可以从输入、模型结构、两阶段流程、知识蒸馏和部署优化几个方向考虑。

### 1. 推理级轻量化

推理级轻量化不改模型权重，只改变推理时送入模型的数据规模。

常用方法：

```bash
--pre-voxel-size 0.05
--fragment-batch-size 8
--point-max 100000
--merge-mode halo
--context-margin 8
```

含义：

- `pre-voxel-size`：推理前先对点云降采样，减少模型真正处理的点数。
- `fragment-batch-size`：一次送入 GPU 的 fragment 数量，通常只影响速度和显存，不影响精度。
- `point-max`：每个 fragment 允许的最大点数，越大上下文越完整，但显存占用越高。
- `merge-mode halo`：推理时看周围上下文，只写回核心区域，适合输电线路这种连续结构。

推荐优先尝试：

```bash
--merge-mode halo \
--tile-size 40 \
--tile-stride 40 \
--context-margin 8 \
--pre-voxel-size 0.05 \
--fragment-batch-size 8
```

如果速度仍然不够，可以逐步尝试：

```bash
--pre-voxel-size 0.06
--fragment-batch-size 12
--point-max 120000
```

注意：`pre-voxel-size` 太大会影响电线和绝缘子这类细小目标，不建议一开始就设置到 `0.10`。

### 2. 输入级轻量化

输入级轻量化是从数据层面减少点数。

常见方法：

- 增大训练和推理中的 `grid_size`，例如从 `0.05` 改成 `0.06`。
- 对纯背景区域做更强的降采样。
- 对杆塔、电线、绝缘子附近区域保留更高点密度。
- 先用传统方法或规则过滤明显无关的地面、远处背景点。
- 按线路走廊裁剪，只保留输电线路附近区域。

对于输电线路场景，不建议全局盲目增大 `grid_size`，因为电线和绝缘子比较细。更合理的方式是：

```text
背景区域可以稀疏一点；
杆塔、电线、绝缘子附近区域保留更密。
```

### 3. 结构级轻量化

结构级轻量化是真正修改模型结构，需要重新训练。

当前 PTv3 模型大致是：

```python
enc_depths=(2, 2, 2, 6, 2)
enc_channels=(32, 64, 128, 256, 512)
dec_depths=(2, 2, 2, 2)
dec_channels=(64, 64, 128, 256)
```

可以设计一个中等轻量版本：

```python
enc_depths=(1, 2, 2, 4, 1)
enc_channels=(24, 48, 96, 192, 384)
dec_depths=(1, 1, 1, 1)
dec_channels=(48, 48, 96, 192)
backbone_out_channels=48
```

也可以设计一个更小的版本：

```python
enc_depths=(1, 1, 2, 3, 1)
enc_channels=(16, 32, 64, 128, 256)
dec_depths=(1, 1, 1, 1)
dec_channels=(32, 32, 64, 128)
backbone_out_channels=32
```

影响：

- 通道数变小：参数量、显存、计算量都会下降。
- 深度变小：推理更快，但表达能力下降。
- 模型太小：绝缘子、小电线、线塔连接处更容易掉精度。

对当前任务，更推荐先做中等轻量版本，不要一开始压得太狠。

### 4. 知识蒸馏

知识蒸馏是比较适合当前项目的轻量化方法。

思路：

```text
大模型 model_best.pth 作为 teacher；
轻量小模型作为 student；
训练时让 student 同时学习真实标签和 teacher 的输出概率。
```

训练损失可以设计为：

```text
总损失 = 真实标签 CE/Lovasz Loss + teacher/student logits 的 KL Loss
```

优点：

- 小模型直接训练时，绝缘子和电线可能掉精度。
- 蒸馏后，小模型可以学习大模型对边界、细线、小目标的判断。
- 比单纯缩小模型更稳。

建议实验对比：

```text
原始大模型：精度、速度、显存
小模型直接训练：精度、速度、显存
小模型蒸馏训练：精度、速度、显存
```

重点观察：

```text
line 的连续性；
insulator 的召回率；
tower 和 background 的误分情况。
```

### 5. 两阶段轻量化

两阶段轻量化非常适合输电线路场景。

核心思想：

```text
第一阶段：轻量模型跑全图，快速分出背景、杆塔、电线和粗略绝缘子候选；
第二阶段：原始大模型只在疑似绝缘子、杆塔顶部、线塔连接区域附近精细推理。
```

原因：

```text
背景：面积大，形态稳定，容易分；
杆塔：结构大，竖直明显，相对容易分；
电线：线状连续结构，比较明显；
绝缘子：小、稀少、靠近杆塔和电线，是最难类别。
```

所以没必要让大模型对整幅 LAS 的所有点都精细推理。更合理的是：

```text
背景、杆塔、电线交给轻量模型；
绝缘子和线塔连接区域交给原始大模型。
```

两阶段流程：

```text
1. 轻量模型全图推理，得到粗分割结果。
2. 根据杆塔点、电线点、绝缘子候选点生成 ROI。
3. 对每个 ROI 扩大一定范围，避免漏掉绝缘子。
4. 使用原始大模型只推理 ROI。
5. 把 ROI 内的大模型结果覆盖回第一阶段结果。
```

ROI 可以选择：

- 杆塔顶部附近。
- 杆塔框周围扩大几米。
- 电线与杆塔接近区域。
- 第一阶段预测为绝缘子的区域。

融合规则示例：

```text
非 ROI 区域：保留轻量模型结果；
ROI 区域：使用大模型结果；
如果大模型预测为绝缘子，优先覆盖；
如果大模型修正了杆塔和电线连接处，也可以覆盖。
```

这个方法的关键是 ROI 不能漏。
第一阶段宁可多召回一些候选区域，也不要过于严格。

### 6. 动态 tile 和密度自适应分块

当前脚本通常使用固定大小的 tile，例如：

```bash
--tile-size 40
--tile-stride 40
```

但实际 LAS 点云密度并不均匀，有些区域点很多，有些区域很稀疏。可以做动态 tile：

```text
点密度低的区域：使用更大的 tile；
点密度高的区域：使用更小的 tile；
纯背景稀疏区域：跳过或粗推；
杆塔、电线区域：保留更高精度。
```

优点：

- 避免某些超密 tile 推理特别慢。
- 避免稀疏区域被切得太碎。
- 更适合长距离输电线路点云。

### 7. 后处理辅助轻量化

后处理不能替代模型，但可以减少模型对某些规则性问题的压力。

当前输电线路可以利用的规则：

- 杆塔通常是竖直连通结构。
- 电线通常是连续线状结构。
- 绝缘子通常靠近杆塔和电线连接处。
- 背景误分成杆塔时，通常高度、连通性、线塔接触关系不满足要求。
- 没有线路接触的孤立杆塔候选可以删除。

因此可以用后处理处理：

```text
背景误分杆塔；
断裂电线连接；
孤立杆塔删除；
杆塔框和档距框拟合；
绝缘子必须靠近杆塔和电线。
```

这样可以让第一阶段模型更轻，同时用规则兜住一部分错误。

### 8. FP16、ONNX、TensorRT 和 INT8

这些属于部署工程方向。

FP16：

```text
使用半精度推理，通常可以降低显存并提升速度。
```

但是需要测试 `spconv`、`torch_scatter` 等算子是否稳定。

ONNX / TensorRT：

```text
理论上可以加速部署，但 PTv3 里有稀疏卷积、动态索引、torch_scatter、序列化逻辑，导出难度较大。
```

当前不建议优先死磕 ONNX/TensorRT，因为之前导出 ONNX 已经遇到动态算子和 PyTorch tracing 问题。

INT8：

```text
可以进一步减少模型体积和计算量，但对点云小目标可能影响明显。
```

尤其是绝缘子这种小类别，INT8 量化可能导致召回率下降。建议后期再做。

### 9. 推荐优先级

当前项目更推荐按下面顺序推进：

```text
第一优先级：pre-voxel + halo 参数优化
第二优先级：两阶段 ROI 推理
第三优先级：轻量 PTv3 student + 知识蒸馏
第四优先级：动态 tile / 密度自适应分块
第五优先级：FP16 推理
第六优先级：ONNX / TensorRT / INT8
```

不建议当前优先做：

```text
非结构化剪枝；
直接 INT8 量化；
强行 ONNX 导出；
一上来把模型通道和深度压得过小。
```

最终比较适合输电线路点云的完整路线是：

```text
轻量 student 模型
+ teacher 知识蒸馏
+ ROI 二阶段精推
+ 动态分块
+ 后处理规则约束
```

这样可以在尽量保证电线、杆塔、绝缘子精度的同时，降低整体推理时间和显存占用。
