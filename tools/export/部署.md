# Pointcept 输电线路模型 Windows 部署方式说明

本文档整理将当前 Pointcept 输电线路点云分割代码部署到 Windows 平台上的几种方式。

当前项目核心内容：

```text
模型权重：
exp/transmission_line/ptv3-4cls-ins-oversample_v2/model/model_best.pth

推理脚本：
tools/infer/infer_las_semseg.py

后处理脚本：
tools/infer/postprocess_tower_line_clean.py
tools/infer/draw_tower_line_boxes.py
```

Windows 目标环境示例：

```text
Windows 10 / Windows 11
NVIDIA 显卡
CUDA 12.1
PyTorch 2.1.0
RTX 4050 或其他 NVIDIA GPU
```

---

## 1. 方式一：Windows 本机 Conda 环境运行

这是当前最推荐的方式。

这种方式不是把模型转成 ONNX 或 TensorRT，而是在 Windows 上安装 Pointcept 所需依赖，直接用 PyTorch 加载 `.pth` 权重推理。

### 优点

- 最接近 Linux 训练环境。
- 精度和原始模型一致。
- 不需要处理 ONNX 导出失败、TensorRT 插件、动态算子等复杂问题。
- 适合 Pointcept/PTv3 这种包含 `spconv`、`torch_scatter`、动态索引的点云模型。

### 缺点

- 需要在 Windows 上配置 Python、Conda、PyTorch、spconv、torch_scatter 等环境。
- 交付给后端时，需要保证对方机器也有对应环境。

### 适用场景

```text
开发调试；
笔记本本地推理；
内部演示；
后端机器允许安装 Conda 环境。
```

### 基本流程

1. 安装 Anaconda 或 Miniconda。
2. 创建 `pointcept` 环境。
3. 安装 PyTorch GPU 版本。
4. 安装 `spconv`、`torch_scatter`、`laspy`、`scipy`、`timm`、`addict`、`yapf`、`regex` 等依赖。
5. 将项目代码、配置文件、权重文件复制到 Windows。
6. 执行推理脚本。

### 环境检查命令

```bat
conda activate pointcept

python -c "import torch; print(torch.__version__); print(torch.version.cuda); print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0) if torch.cuda.is_available() else 'no cuda')"

python -c "import spconv.pytorch as spconv; print('spconv ok')"

python -c "import torch_scatter; print('torch_scatter ok')"

python -c "import laspy; print('laspy ok')"

python -c "import scipy; print('scipy ok')"

python -c "import addict, timm, yapf, regex; print('basic deps ok')"
```

### Windows 推理命令示例

```bat
conda activate pointcept

cd /d D:\6_24_demo\pointCeptV1.5-lidar

python tools\infer\infer_las_semseg.py ^
  --input D:\data\cloudb009b5736892392a.las ^
  --output D:\data\cloudb_pred.las ^
  --merge-mode halo ^
  --tile-size 40 ^
  --tile-stride 40 ^
  --context-margin 10 ^
  --pre-voxel-size 0.05 ^
  --fragment-batch-size 1 ^
  --min-tile-points 1024 ^
  --disable-flash ^
  --overwrite
```

说明：

- Windows 上如果没有 `flash-attn`，建议加 `--disable-flash`。
- 如果显存足够，可以把 `--fragment-batch-size` 从 `1` 提高到 `2`、`4`、`8`。
- RTX 4050 笔记本显存有限，建议先从 `1` 或 `2` 开始。

## 2. 方式二：打包 Conda 环境后交付

这种方式是在一台已经配置好的 Windows 机器上，把整个 Conda 环境打包，然后复制到另一台 Windows 机器上使用。

### 优点

- 后端或使用方不需要从头安装大量依赖。
- 环境更一致，减少版本不匹配。
- 比单纯写安装文档更稳定。

### 缺点

- 环境包通常比较大。
- 目标机器仍然需要 NVIDIA 驱动。
- CUDA、PyTorch、显卡驱动版本需要兼容。
- 路径变化时，某些环境可能需要重新修复。

### 适用场景

```text
同类型 Windows 机器批量部署；
内部交付；
不想让使用方手动安装复杂依赖。
```

### 推荐工具

可以使用 `conda-pack`：

```bat
conda activate base
pip install conda-pack

conda pack -n pointcept -o pointcept_env.zip
```

在目标 Windows 机器上：

```bat
mkdir D:\pointcept_env
tar -xf pointcept_env.zip -C D:\pointcept_env

D:\pointcept_env\Scripts\activate.bat
python -c "import torch; print(torch.cuda.is_available())"
```

如果环境提示路径需要修复，可以执行：

```bat
D:\pointcept_env\Scripts\conda-unpack.exe
```

### 注意事项

这种方式并不是生成一个单独的 `.exe`，而是把 Python 环境整体搬过去。  
它的本质仍然是 PyTorch + Pointcept 推理，只是减少了环境安装步骤。

## 3. 方式三：封装成命令行工具给后端调用

这种方式不改变模型和环境，只是在 Windows 上提供一个固定命令或 `.bat` 脚本，让后端通过命令行调用。

### 优点

- 实现简单。
- 稳定性高。
- 后端只需要传入输入路径和输出路径。
- 适合先落地工程流程。

### 缺点

- 仍然依赖 Conda/Python 环境。
- 并不是单文件部署。

### 适用场景

```text
后端只需要调用推理；
不需要 Python API；
可以接受服务器上安装 Conda 环境。
```

### bat 脚本示例

可以创建 `run_infer.bat`：

```bat
@echo off
set INPUT=%1
set OUTPUT=%2

call conda activate pointcept
cd /d D:\6_24_demo\pointCeptV1.5-lidar

python tools\infer\infer_las_semseg.py ^
  --input "%INPUT%" ^
  --output "%OUTPUT%" ^
  --merge-mode halo ^
  --tile-size 40 ^
  --tile-stride 40 ^
  --context-margin 10 ^
  --pre-voxel-size 0.05 ^
  --fragment-batch-size 1 ^
  --min-tile-points 1024 ^
  --disable-flash ^
  --overwrite
```

后端调用：

```bat
run_infer.bat D:\data\input.las D:\data\output.las
```

这种方式对后端最友好的一点是：后端不用理解 Pointcept，只需要执行命令并等待输出文件生成。

### 方式三具体怎么调用

假设你把脚本保存为：

```text
D:\6_24_demo\pointCeptV1.5-lidar\run_infer.bat
```

那么调用方式就是传入两个参数：

```text
第 1 个参数：输入 LAS 文件路径
第 2 个参数：输出 LAS 文件路径
```

手动测试命令：

```bat
cd /d D:\6_24_demo\pointCeptV1.5-lidar

run_infer.bat D:\data\input.las D:\data\output.las
```

如果路径里有空格，必须加双引号：

```bat
run_infer.bat "D:\my data\input.las" "D:\my data\output.las"
```

推荐把 `run_infer.bat` 写得稍微完整一点，方便后端判断成功或失败：

```bat
@echo off
set INPUT=%~1
set OUTPUT=%~2

if "%INPUT%"=="" (
  echo ERROR: missing input las path
  exit /b 1
)

if "%OUTPUT%"=="" (
  echo ERROR: missing output las path
  exit /b 1
)

if not exist "%INPUT%" (
  echo ERROR: input file not found: "%INPUT%"
  exit /b 2
)

call conda activate pointcept
if errorlevel 1 (
  echo ERROR: conda activate pointcept failed
  exit /b 3
)

cd /d D:\6_24_demo\pointCeptV1.5-lidar

python tools\infer\infer_las_semseg.py ^
  --input "%INPUT%" ^
  --output "%OUTPUT%" ^
  --merge-mode halo ^
  --tile-size 40 ^
  --tile-stride 40 ^
  --context-margin 10 ^
  --pre-voxel-size 0.05 ^
  --fragment-batch-size 1 ^
  --min-tile-points 1024 ^
  --disable-flash ^
  --overwrite

if errorlevel 1 (
  echo ERROR: inference failed
  exit /b 4
)

echo OK: "%OUTPUT%"
exit /b 0
```

后端只需要判断进程退出码：

```text
0：成功
1：缺少输入或输出参数
2：输入文件不存在
3：Conda 环境激活失败
4：推理脚本执行失败
```

### 后端调用示例

Python 调用：

```python
import subprocess

cmd = [
    r"D:\6_24_demo\pointCeptV1.5-lidar\run_infer.bat",
    r"D:\data\input.las",
    r"D:\data\output.las",
]

result = subprocess.run(cmd, capture_output=True, text=True)

print(result.stdout)
print(result.stderr)

if result.returncode != 0:
    raise RuntimeError(f"infer failed, code={result.returncode}")
```

Java 调用：

```java
ProcessBuilder pb = new ProcessBuilder(
    "D:\\6_24_demo\\pointCeptV1.5-lidar\\run_infer.bat",
    "D:\\data\\input.las",
    "D:\\data\\output.las"
);
pb.redirectErrorStream(true);
Process process = pb.start();
int code = process.waitFor();
if (code != 0) {
    throw new RuntimeException("infer failed, code=" + code);
}
```

C# 调用：

```csharp
using System.Diagnostics;

var psi = new ProcessStartInfo
{
    FileName = @"D:\6_24_demo\pointCeptV1.5-lidar\run_infer.bat",
    Arguments = "\"D:\\data\\input.las\" \"D:\\data\\output.las\"",
    UseShellExecute = false,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    CreateNoWindow = true
};

var process = Process.Start(psi);
process.WaitForExit();

if (process.ExitCode != 0)
{
    throw new Exception("infer failed, code=" + process.ExitCode);
}
```

Node.js 调用：

```javascript
const { spawn } = require("child_process");

const child = spawn(
  "D:\\6_24_demo\\pointCeptV1.5-lidar\\run_infer.bat",
  ["D:\\data\\input.las", "D:\\data\\output.las"],
  { shell: true }
);

child.stdout.on("data", data => console.log(data.toString()));
child.stderr.on("data", data => console.error(data.toString()));

child.on("close", code => {
  if (code !== 0) {
    throw new Error(`infer failed, code=${code}`);
  }
});
```

### 调用时要注意

- 后端传入的路径必须是 Windows 本机能访问的路径，例如 `D:\data\input.las`。
- 不建议多个任务同时调用同一张 GPU 推理，容易显存不足；后端最好做单任务队列。
- 输出文件存在时，脚本里已经加了 `--overwrite`，会直接覆盖。
- 如果需要保留日志，可以让后端把 stdout/stderr 写入 `.log` 文件。
- 如果后端服务和推理机器不是同一台，需要先把 LAS 上传到推理机器本地，再调用 bat。

## 4. 方式四：封装成 HTTP 服务

这种方式是在 Windows 机器上启动一个 Python 服务，例如 FastAPI。后端通过 HTTP 请求提交 LAS 文件路径，然后服务内部调用推理脚本。

### 优点

- 后端调用方式统一。
- 可以做任务队列、日志、状态查询。
- 适合多人或多系统调用。
- 方便后续接入 Web 平台。

### 缺点

- 需要维护一个常驻服务。
- 需要处理并发、任务状态、异常、磁盘路径等问题。
- GPU 同时跑多个任务可能爆显存，需要限制队列。

### 适用场景

```text
后端系统集成；
服务器部署；
多人提交点云推理任务；
需要任务状态查询。
```

### 服务接口设计示例

```text
POST /infer
输入：
{
  "input": "D:/data/input.las",
  "output": "D:/data/output.las"
}

输出：
{
  "status": "ok",
  "output": "D:/data/output.las",
  "elapsed_sec": 123.45
}
```

### 注意事项

- 建议一次只允许一个 GPU 推理任务运行。
- 大 LAS 文件不要直接通过 HTTP 上传，可以传文件路径。
- 服务内部仍然调用 `infer_las_semseg.py`。
- 后处理可以作为第二步接在推理完成后执行。

## 5. 方式五：打包成 exe

这种方式是使用 PyInstaller 等工具把 Python 脚本封装成 `.exe`。

### 优点

- 表面上看起来像普通 Windows 程序。
- 后端可以直接调用 `.exe input output`。

### 缺点

- 对 Pointcept 这种项目并不推荐作为第一选择。
- `torch`、`spconv`、`torch_scatter`、CUDA 动态库很多，打包复杂。
- 最终 exe 和依赖文件仍然会非常大。
- 容易出现 DLL 找不到、CUDA 算子加载失败等问题。
- 即使生成 exe，也通常不是单文件干净交付。

### 适用场景

```text
必须以 exe 形式交付；
目标机器环境相对固定；
可以接受 exe 旁边带大量依赖文件。
```

### 推荐理解

对当前项目来说，`.exe` 打包更像是：

```text
把 Python 启动入口包装成 exe
```

而不是：

```text
把 Pointcept 模型真正编译成一个独立程序
```

所以如果只是给后端调用，优先推荐：

```text
bat 命令行方式
或 HTTP 服务方式
```

而不是一开始就打包 exe。

## 6. 方式六：ONNX Runtime 推理

这种方式是把 `.pth` 模型导出为 `.onnx`，然后在 Windows 上使用 ONNX Runtime GPU 推理。

### 优点

- 理论上更适合部署。
- 后端可以不依赖完整 Pointcept 代码。
- ONNX Runtime 在 Windows 上支持较好。

### 缺点

当前 PTv3/Pointcept 模型导出 ONNX 难度较大，因为模型中包含：

```text
spconv
torch_scatter
动态索引
点云序列化逻辑
稀疏卷积
自定义数据结构
```

这些算子和动态逻辑对 ONNX 不友好。

之前导出 ONNX 时已经出现过类似错误：

```text
ONNX export failed
aten::view / alias_analysis / tracing error
```

这说明当前模型不是普通 CNN，不能简单地 `torch.onnx.export()` 后直接部署。

### 适用场景

```text
后续专门做部署研发；
愿意修改模型 forward；
愿意替换或重写不支持 ONNX 的算子；
愿意固定输入点数和推理流程。
```

### 当前建议

当前阶段不建议优先使用 ONNX Runtime 作为主部署方案。  
除非后续专门做模型导出适配，否则应该优先使用 PyTorch 原生推理。

## 7. 方式七：TensorRT 部署

TensorRT 通常需要先导出 ONNX，再构建 TensorRT engine。

### 优点

- 理论推理速度最快。
- FP16/INT8 优化能力强。
- 适合成熟部署阶段。

### 缺点

- 依赖 ONNX 成功导出。
- PTv3 中的稀疏算子和动态索引很可能需要自定义 TensorRT plugin。
- 工程成本高。
- 调试难度大。

### 适用场景

```text
模型结构已经部署友好；
输入尺寸固定；
有专门部署工程时间；
需要极致推理速度。
```

### 当前建议

当前项目不建议优先做 TensorRT。  
更合理的路线是先完成：

```text
Windows PyTorch 推理
两阶段轻量化
小模型训练或蒸馏
```

之后如果仍然需要更高性能，再考虑 TensorRT。

---

## 8. 方式八：WSL2 / Docker 部署

这种方式是在 Windows 上使用 WSL2 或 Docker 跑 Linux 环境。

### 优点

- 更接近 Linux 训练环境。
- Linux 下 Pointcept、spconv、torch_scatter 等依赖通常更好处理。
- 对研发人员比较友好。

### 缺点

- 对普通 Windows 用户不够直观。
- 需要配置 WSL2、NVIDIA Container Toolkit 或 Docker GPU。
- 给后端交付时不如原生 Windows 直接。

### 适用场景

```text
研发测试；
Windows 机器但希望复用 Linux 环境；
部署人员熟悉 Docker/WSL2。
```

### 当前建议

如果目标是给普通后端系统调用，优先考虑 Windows 原生 Conda 或 HTTP 服务。  
如果目标是研发复现，WSL2/Docker 是不错的选择。

## 9. 推荐部署路线

结合当前项目情况，推荐优先级如下：

```text
第一推荐：Windows 本机 Conda + PyTorch 原生推理
第二推荐：Conda 环境打包交付
第三推荐：bat 命令行封装给后端调用
第四推荐：FastAPI/HTTP 服务封装
第五推荐：exe 包装入口
第六推荐：ONNX Runtime
第七推荐：TensorRT
第八推荐：WSL2/Docker
```

如果目标是“尽快让 Windows 笔记本能跑起来”：

```text
选择方式一：Windows 本机 Conda 环境运行。
```

如果目标是“交付给后端调用”：

```text
优先选择方式三：bat 命令行工具；
或者方式四：HTTP 服务。
```

如果目标是“减少使用方安装环境的麻烦”：

```text
选择方式二：打包 Conda 环境。
```

如果目标是“真正工业级高性能部署”：

```text
后续再研究 ONNX/TensorRT，但当前不是最高优先级。
```

## 10. 当前最稳的落地方案

当前阶段建议采用：

```text
Windows Conda 环境
+ PyTorch 原生加载 model_best.pth
+ infer_las_semseg.py 命令行推理
+ postprocess_tower_line_clean.py 后处理
+ draw_tower_line_boxes.py 可视化画框
```

完整流程：

```bat
conda activate pointcept

cd /d D:\6_24_demo\pointCeptV1.5-lidar

python tools\infer\infer_las_semseg.py ^
  --input D:\data\input.las ^
  --output D:\data\pred.las ^
  --merge-mode halo ^
  --tile-size 40 ^
  --tile-stride 40 ^
  --context-margin 10 ^
  --pre-voxel-size 0.05 ^
  --fragment-batch-size 1 ^
  --min-tile-points 1024 ^
  --disable-flash ^
  --overwrite

python tools\infer\postprocess_tower_line_clean.py ^
  --input D:\data\pred.las ^
  --output D:\data\clean.las ^
  --overwrite

python tools\infer\draw_tower_line_boxes.py ^
  --input D:\data\clean.las ^
  --output D:\data\clean_boxes.las ^
  --overwrite
```

这条路线的优点是：

```text
精度最接近 Linux 原始模型；
不需要 ONNX/TensorRT；
遇到问题容易排查；
后续可以自然封装成 bat 或 HTTP 服务。
```
