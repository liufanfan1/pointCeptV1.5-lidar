# 输电线路点云推理与后处理

## 1. LAS 推理脚本

新增脚本：

```text
tools/infer_las_semseg.py
```

用途：

- 读取单个 `.las` 文件或目录内多个 `.las` 文件。
- 加载权重：

```text
exp/transmission_line/ptv3-4cls-ins-oversample_v2/model/model_best.pth
```

- 使用对应配置：

```text
exp/transmission_line/ptv3-4cls-ins-oversample_v2/config.py
```

- 输出带预测结果的 `.las`。
- 将预测类别写入 LAS 的 `classification` 字段。
- 默认把分割结果写成 RGB 颜色，方便 CloudCompare 直接可视化。

类别和颜色：

| ID | 类别 | 显示颜色 |
| --- | --- | --- |
| 0 | ground | 灰色 |
| 1 | tower | 红色 |
| 2 | line | 蓝色 |
| 3 | insulator | 黄色 |

## 2. 原始测试流程

原始 `tools/test.py` 并不是直接读取整幅 `.las` 推理，而是读取预处理后的 `.pth` tile。

原测试流程：

```text
LAS 分类文件
  -> preprocess_transmission_line.py
  -> data/.../test/*.pth
  -> tools/test.py
  -> result/*_pred.npy
```

`.pth` tile 内字段主要包括：

```text
coord
color
semantic_gt
origin
scene
```

测试配置内的核心 transform：

```text
CenterShift(apply_z=True)
NormalizeColor
GridSample(grid_size=0.05, mode=test)
SphereCrop(point_max=100000, mode=all)
RandomScale(scale=[1,1])
CenterShift(apply_z=False)
ToTensor
Collect(coord, grid_coord, index, feat=coord+color)
```

模型输入特征：

```text
feat = coord + color
```

原始测试代码中 `fragment_batch_size = 1`，每个 fragment 推理后将 softmax 概率累加回原 tile 点索引，最后 `argmax` 得到类别。

## 3. LAS 推理脚本与原流程的对应关系

`tools/infer_las_semseg.py` 为了支持直接输入 LAS，额外补了外层处理：

```text
读取 LAS
  -> 坐标减 origin / 转局部坐标
  -> 按 XY 切 tile
  -> 逐 tile 调用原测试 pipeline
  -> 写回 LAS classification / RGB
```

关键修正：

- 不能将大地坐标直接转 `float32`。
- 必须先做局部坐标转换：

```python
coord = coord - coord.min(axis=0)
coord = coord.astype(np.float32)
```

原因：

- 输入 LAS 坐标可能是 `x=526000, y=3514000` 这种大坐标。
- 如果直接转 `float32`，5cm 网格精度会受影响。
- 这会导致 `GridSample/SphereCrop` 行为偏离原始 `.pth` 测试流程，fragment 数量和耗时异常。

## 4. 推理性能问题与原因

测试 LAS：

```text
/24085403037/24085403037/shixi/dataset/6_23_demo/cloudb009b5736892392a.las
```

文件信息：

```text
大小: 约 1.56GB
点数: 48,095,161
LAS 版本: 1.2
point format: 3
```

最初推理慢的主要原因：

1. 直接把整幅 4800 万点云送入原测试流程，`SphereCrop(mode=all)` 极慢。
2. 按 tile 推理后，部分 tile 点数超过 100 万，仍然很慢。
3. 早期脚本每个 tile 都扫描全量 4800 万点，已改为非重叠 tile 一次性分桶。
4. fallback 写 LAS 早期是逐点 `seek/write`，已改为 `np.memmap` 向量化写入。
5. 坐标未先减 origin 会导致 fragment 行为异常，已修正。

原测试日志对比：

```text
1,206,530 点 tile -> 约 122s
1,357,965 点 tile -> 约 23.8s
3,936,550 点 tile -> 约 137s
```

这说明模型原始流程本身不应出现单个百万点 tile 运行 1800s 的情况；如果出现，通常是输入预处理和原流程不一致。

## 5. 当前推理模式

脚本现在支持三种 tile 合并模式。

### 5.1 plain

当前最基础模式：

```text
非重叠 tile
每个点只由一个 tile 预测
速度最快
边界上下文最少
```

示例：

```bash
python tools/infer_las_semseg.py \
  --input input.las \
  --output output_plain.las \
  --merge-mode plain \
  --tile-size 40 \
  --tile-stride 40 \
  --pre-voxel-size 0.05 \
  --overwrite
```

### 5.2 halo

推荐优先尝试：

```text
核心 tile 仍是 40m
推理时向周围扩展 context-margin
只写回中心核心区
边界处有上下文
比 overlap 快
```

示例：

```bash
python tools/infer_las_semseg.py \
  --input input.las \
  --output output_halo.las \
  --merge-mode halo \
  --tile-size 40 \
  --tile-stride 40 \
  --context-margin 10 \
  --pre-voxel-size 0.05 \
  --overwrite
```

### 5.3 overlap

最稳但最慢：

```text
tile 有重叠
同一个点可能被多个 tile 推理
累加 softmax 概率
最终取平均概率 argmax
```

示例：

```bash
python tools/infer_las_semseg.py \
  --input input.las \
  --output output_overlap.las \
  --merge-mode overlap \
  --tile-size 40 \
  --tile-stride 20 \
  --pre-voxel-size 0.05 \
  --overwrite
```

## 6. 当前推荐推理参数

速度和效果折中：

```bash
python tools/infer_las_semseg.py \
  --input /path/to/input.las \
  --output /path/to/output_vis.las \
  --merge-mode halo \
  --tile-size 40 \
  --tile-stride 40 \
  --context-margin 10 \
  --pre-voxel-size 0.05 \
  --fragment-batch-size 1 \
  --min-tile-points 1024 \
  --overwrite
```

更接近原始测试流程：

```bash
python tools/infer_las_semseg.py \
  --input /path/to/input.las \
  --output /path/to/output_strict.las \
  --merge-mode plain \
  --tile-size 40 \
  --tile-stride 40 \
  --fragment-batch-size 1 \
  --min-tile-points 1024 \
  --overwrite
```

注意：

- 不加 `--pre-voxel-size` 更接近原测试流程，但大 LAS 会慢。
- `--pre-voxel-size 0.05` 与模型 `GridSample(grid_size=0.05)` 同尺度，通常是合理的速度优化。
- 如果追求更细节，可试 `--pre-voxel-size 0.03`。

## 7. 推理批次大小对精度的影响

`--fragment-batch-size` 只控制一次送几个 fragment 到 GPU。

通常：

```text
不会明显影响精度
主要影响显存和速度
```

如果严格复现原始 `tools/test.py`：

```bash
--fragment-batch-size 1
```

显存充足时可试：

```bash
--fragment-batch-size 2
```

或：

```bash
--fragment-batch-size 4
```

如果 OOM，再降回 1。

## 8. CloudCompare 可视化

新版推理脚本默认将预测类别写成 RGB。

输出 LAS 打开后应能看到：

```text
ground     灰色
tower      红色
line       蓝色
insulator  黄色
```

如果 CloudCompare 看不到颜色：

1. 选中点云。
2. 切到 RGB colors 显示。
3. 或查看 `Scalar fields -> Classification`。

如果不想覆盖原始 RGB，可在推理时加：

```bash
--no-colorize
```

## 9. 净空距离后处理脚本

新增脚本：

```text
tools/postprocess_line_clearance_box.py
```

用途：

- 读取分割后的 LAS。
- 找电线类 `line_class=2` 到指定背景类别的最近距离。
- 输出 JSON 报告。
- 在 LAS 中高亮最近区域。
- 追加 3D 框线点，方便 CloudCompare 查看。

示例：

```bash
python tools/postprocess_line_clearance_box.py \
  --input /path/to/cloudb_outPut.las \
  --output /path/to/cloudb_clearance_box.las \
  --report /path/to/cloudb_clearance_report.json \
  --background-classes 0 \
  --coarse-voxel-size 1.0 \
  --refine-radius 3.0 \
  --box-margin 2.0 \
  --overwrite
```

其中：

```text
--background-classes 0
```

表示只计算电线到 `ground` 的最近距离。

如果当前模型没有“树木”类别，就不能严格计算“电线到树木”的距离，只能计算电线到已有分类的距离。

## 10. 净空报告含义

示例报告中的关键字段：

```json
"nearest_distance_m": 0.08473653302882522
```

表示电线点到指定背景点的最近距离约为：

```text
8.47 cm
```

其他字段：

```text
line_point_index
background_point_index
```

最近的电线点和背景点在 LAS 中的点索引。

```text
line_point_xyz
background_point_xyz
```

最近点坐标。

```text
box_min_xyz
box_max_xyz
```

3D 框范围。

```text
points_inside_box
```

框内原始点数量。

```text
refined_candidate_counts
```

粗搜索后，局部精修实际参与精确计算的点数。

注意：

- 如果 `nearest_distance_m` 很小，可能是真实风险，也可能是分割误差或噪声。
- 应在 CloudCompare 中查看输出的 `*_clearance_box.las`，确认框内是否是真实风险区域。

## 11. 后处理性能优化

原始最近距离脚本用：

```bash
--search-voxel-size 0.10
```

对 3300 万 ground 点体素化后仍有约 1400 万代表点，因此很慢。

已改为两阶段搜索：

```text
1. coarse voxel 全局粗搜索
2. 局部原始点精修
```

推荐：

```bash
--coarse-voxel-size 1.0
--refine-radius 3.0
```

更快：

```bash
--coarse-voxel-size 2.0
```

更细：

```bash
--coarse-voxel-size 0.5
```

## 12. 后续改进方向

当前后处理是“全局粗搜 + 局部精修”。

对于电力巡检业务，更合理的后续方向是：

```text
1. 提取电线点
2. 拟合线路方向或线路段
3. 沿导线生成 corridor 管廊
4. 只保留管廊内背景点
5. 计算导线到障碍物净空距离
6. 输出风险框和报告
```

这样比全场搜索更快，也更符合业务语义。
